2. Context retrieval — filter function and what gets sent
The filter function is retrieveContext(userMessage, feature) in contextRetrieval.ts. It uses two helpers:

function hasKeyword(message: string, keywords: string[]): boolean {
  const lower = message.toLowerCase();
  return keywords.some((k) => lower.includes(k.toLowerCase()));
}
function featureIs(feature: string | undefined, ...names: string[]): boolean {
  if (!feature) return false;
  const f = feature.toLowerCase();
  return names.some((n) => f.includes(n.toLowerCase()));
}

Each data category has two independent triggers — the active UI feature or keywords in the message — so the right context is returned even if the feature flag is wrong:

Category	Feature trigger	Keyword examples
Emergency	emergency	hurt, medical, fire, evacuate, cardiac …
Accessibility	accessibility	wheelchair, lift, ramp, disabled …
Navigation	navigation	gate, section, where, find, direction …
Food	food, dining	hungry, eat, vegan, halal, gluten …
Transport	transport	metro, bus, taxi, shuttle, train …
Sustainability	sustainability	water, refill, recycle …
Lost & Found	lost	missing, backpack, passport, child …
Announcements	announce, announcement	(feature-only)
Fallback	(nothing matched)	Brief stadium summary
What gets sent to Gemini — raw JSON is never sent. The function maps each data slice to a hand-formatted plain-text string:

// Example — navigation query: "Where is Gate A?"
// contextBlocks[0].data:
`STADIUM GATES (current crowd levels):
- Gate A (North): crowd=low, open=true
- Gate B (East): crowd=medium, open=true
- Gate C (South): crowd=high, open=true
- Gate D (West): crowd=medium, open=true
- Gate E (South-East): crowd=high, open=false
STADIUM: Lusail Iconic Stadium, Capacity: 88,966
Gate layout: Gate A, Gate B, Gate C, Gate D, Gate E`

Fields like id, floor, categories, and other internal properties are deliberately omitted. The formatted strings are collected and passed to buildSystemPrompt as contextBlocks.map((c) => c.data) — the system prompt receives plain text only.

The selection labels are also logged via req.log.info({ labels: contextBlocks.map((c) => c.label) }, "ArenaOS context retrieval selected") for observability without logging the data itself.

3. Stadium data — module-level load, not per-request
Confirmed. stadiumData.ts declares all data as top-level export const literals:

export const gates = [ { id: 'gate-a', name: 'Gate A', ... }, ... ];
export const foodCourts = [ ... ];
export const transportOptions = [ ... ];
export const accessibilityFeatures = [ ... ];
export const emergencyPoints = [ ... ];
export const sustainabilityPoints = [ ... ];
export const stadium = { name: 'Lusail Iconic Stadium', capacity: 88966, ... };

Both consumers import these at the module level:

// contextRetrieval.ts — top of file
import { gates, foodCourts, transportOptions, ... } from "./stadiumData.js";
// organizer.ts — top of file
import { gates, foodCourts, transportOptions, emergencyPoints } from "../../data/stadiumData.js";

Node's module system evaluates each module exactly once and caches the exports. Every request hits the same already-evaluated objects — no re-reading, no re-parsing, no per-request allocation of the data arrays.
